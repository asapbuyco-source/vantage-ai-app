# Vantage AI — System Audit Report
**Date:** 2026-06-27
**Auditor:** Senior Software Architect
**Scope:** Full-stack application (Frontend, Backend, Python Quant Engine, Cloud Infrastructure)

---

## 1. SYSTEM OVERVIEW

**Purpose:** Vantage AI is a production-grade sports betting prediction SaaS targeting the Cameroonian market. It uses quantitative models (Poisson, Elo, Form-based) and AI (Google Gemini, OpenAI) to generate football, basketball, and cricket betting predictions. It offers free predictions and a paid VIP ("Alpha") tier with accumulators, vault simulation, and arbitrage detection.

**Key Components:**

| Layer | Technology | Role |
|-------|-----------|------|
| Frontend | React 19 + TypeScript 5.8 + Vite 6.4 | PWA SPA, HashRouter for Capacitor, lazy-loaded pages |
| State Management | React Context (AppContext, AuthContext, DataContext) | Theme, language, auth, prediction cache |
| Backend API | Express 5.2.1 (Node.js, ESM) | Gemini/OpenAI proxy, payment webhooks, SSR, admin triggers, sitemap |
| Firestore | Firebase Firestore | User profiles, predictions, blogs, payment intents, vault data |
| Auth | Firebase Auth + custom Firebase Admin JWT | Browser auth (Google/Email), admin JWT for server endpoints |
| Quant Engine | Python 3.11 (50+ modules) | Statistical pipeline: data fetch -> Poisson/Elo/Form -> EV -> risk filter -> Kelly staking -> save |
| Scheduler | node-cron (server-side) | 10 cron jobs at Africa/Lagos times with distributed Firestore locks |
| Payments | Fapshi (MoMo), Selar (cards), RevenueCat (Google Play) | Multi-gateway VIP subscription fulfillment |
| CI/CD | Netlify (frontend), Railway (backend), Codemagic (Android) | Automated deploys, Android AAB/APK builds |

## 2. ISSUES

### ISSUE-001: RevenueCat Webhook Authentication is Optional (Bypassable)
- **ID:** SEC-001
- **Description:** The RevenueCat webhook handler at `server.js:834` conditionally verifies the `Authorization` header only when `REVENUECAT_WEBHOOK_SECRET` is a non-empty string. If the environment variable is unset or empty, **any** unauthenticated request can trigger VIP fulfillment.
- **Root Cause:** The guard clause `if (REVENUECAT_WEBHOOK_SECRET)` on line 836 wraps the auth check, making it skippable. Missing a hard assertion that the secret must be configured for this endpoint to be operational.
- **Severity:** **Critical**
- **Location:** `server.js:834-841`
- **Evidence:**
  ```js
  const REVENUECAT_WEBHOOK_SECRET = process.env.REVENUECAT_WEBHOOK_SECRET || '';
  if (REVENUECAT_WEBHOOK_SECRET) {
      const authHeader = req.headers['authorization'] || '';
      if (!authHeader || authHeader !== `Bearer ${REVENUECAT_WEBHOOK_SECRET}`) {
          return res.status(401).json({ error: 'Unauthorized' });
      }
  }
  // If REVENUECAT_WEBHOOK_SECRET is '', auth is entirely skipped.
  ```

### ISSUE-002: Payment Plans Test-Implementation Mismatch (Stale Tests)
- **ID:** TEST-001
- **Description:** `backend/paymentPlans.test.js` expects amounts in XAF (500, 2000, 5000, 12000, 35000), but the source file `backend/paymentPlans.js` defines amounts in USD (4.99, 14.99, 24.99, 59.99, 99.99). Every assertion in the test file will fail. These tests have been stale since the "global pricing (USD)" commit `6399432`.
- **Root Cause:** Amounts were changed from XAF to USD without updating the corresponding unit tests. No CI/CD gate detected the test failure (tests likely not enforced in the build pipeline).
- **Severity:** **High**
- **Location:** `backend/paymentPlans.test.js:15-28` vs `backend/paymentPlans.js:2-8`

### ISSUE-003: Admin Trigger Endpoints Missing Rate Limiting
- **ID:** SEC-002
- **Description:** Admin trigger endpoints (`/api/admin/generate-blog`, `/api/admin/generate-accumulators`, `/api/admin/telegram-broadcast`, `/api/admin/trigger-quant`, `/api/admin/grade-quant`, `/api/admin/quant-performance`) have `adminAuth` middleware but no `rateLimit` middleware. An attacker who obtains admin credentials or a valid admin JWT could flood these endpoints, exhausting API credits (API-Football, OpenAI, Gemini) and Firestore writes.
- **Root Cause:** Rate limiters were applied only to Gemini (`geminiLimiter`, 50/15min) and OpenAI (`openaiLimiter`, 30/15min) proxies, but not to upstream admin trigger endpoints that ultimately invoke those same costly APIs.
- **Severity:** **High**
- **Location:** `server.js:345-474` (all admin POST endpoints from line 345 to 474)

### ISSUE-004: Distributed Lock Fallback on Error Bypasses Locking
- **ID:** REL-001
- **Description:** The `acquireLock` function in `backend/scheduler.js:25-49` returns `true` (granting the lock) when the Firestore transaction errors:
  ```js
  } catch (err) {
      logger.warn({ task, err: err.message }, '[Scheduler] Lock acquisition failed, proceeding anyway');
      return true; // on error, proceed rather than deadlock
  }
  ```
  If Firestore is momentarily unreachable, two Railway instances could simultaneously run the same pipeline, causing duplicate writes, double API credit consumption, and prediction data inconsistencies.
- **Root Cause:** Fail-open design choice for availability over consistency. The `return true` ensures cron jobs always run but removes the safety guarantee of the distributed lock.
- **Severity:** **High**
- **Location:** `backend/scheduler.js:46-48`

### ISSUE-005: Python Binary Resolution Race Condition
- **ID:** REL-002
- **Description:** `resolvePythonBin()` in `backend/quantService.js:42-68` uses a module-level variable `_pythonBin` as a cache. However, the function is `async` — if called twice before the first call resolves (e.g., at module load on line 199 plus an immediate admin trigger), both invocations will probe all Python candidates concurrently, wasting resources. The setter on line 60 (`_pythonBin = candidate`) runs inside the async function body, so concurrent callers see `null` and both start probing.
- **Root Cause:** Cache assignment happens after the await-gated probe, not before. No promise-based deduplication mechanism.
- **Severity:** **Medium**
- **Location:** `backend/quantService.js:40-68`

### ISSUE-006: Fapshi Webhook Amount Validation Uses Wrong Unit
- **ID:** PAY-001
- **Description:** The Fapshi verify endpoint at `server.js:704-708` calls `inferPlanFromAmount(paidAmount)` to map the paid amount to a plan. `paidAmount` comes from the Fapshi response (`data.amount`) which returns XAF (Cameroonian francs). But `PLAN_CONFIG` in `paymentPlans.js` defines amounts in USD (4.99, 14.99, etc.). `inferPlanFromAmount` will fail to match any plan because XAF amounts (e.g., 3500 XAF for ~$5, 15000 XAF for ~$25) are far higher than USD amounts in the config. This means Fapshi VIP payments will always fail verification with "Payment amount does not match selected plan."
- **Root Cause:** Fapshi processes in XAF, but the plan config was migrated from XAF to USD without adding currency-awareness to `inferPlanFromAmount` or adjusting how Fapshi verify maps amounts.
- **Severity:** **High**
- **Location:** `server.js:704-708`, `backend/paymentPlans.js:2-8,26-32`

### ISSUE-007: Multiple Inconsistent Date-Key Implementations
- **ID:** ARCH-001
- **Description:** The Africa/Lagos today date key YYYY-MM-DD is computed in at least four separate locations with slightly different logic:
  - `services/db.ts:35-36` — `toLocaleDateString('en-CA', { timeZone: 'Africa/Lagos' })`
  - `backend/quantService.js:555-557` — same method
  - `backend/scheduler.js:341` — same method in `repairCorruptedPredictions`
  - `backend/blogGenerator.js:3-10` — manual computation: `localMs = now.getTime() + (lagosOffset - now.getTimezoneOffset()) * 60000`
  - `backend/telegramService.js:23-30` — manual computation (different formula)
  
  The manual computations in blogGenerator and telegramService differ from `toLocaleDateString` and each other. At certain times of day near midnight UTC, these will produce different date keys, causing blog/telegram jobs to miss or duplicate content.
- **Root Cause:** No shared utility for date-key computation. Each module re-implements independently.
- **Severity:** **Medium**
- **Location:** See paths above.

### ISSUE-008: Service Worker Version Stale Between Builds
- **ID:** OPS-001
- **Description:** `public/sw.js:15` hardcodes `const CACHE_VERSION = 'v3-' + '20260520';`. The `swVersionPlugin` in `vite.config.ts:14-26` does a string-replace at build time, but the fallback date `20260520` is already 37 days old at the time of this audit. If the replace fails (e.g., regex mismatch, file not found, SW modified with different format), the old version persists and causes stale cache issues for all users.
- **Root Cause:** The replacement regex `const CACHE_VERSION = ['"].*?['"];` relies on exact format match. If `public/sw.js` is edited independently of the plugin, the replace fails silently.
- **Severity:** **Medium**
- **Location:** `public/sw.js:15`, `vite.config.ts:14-26`

### ISSUE-009: No Input Validation on Gemini/OpenAI Proxy Request Bodies
- **ID:** SEC-003
- **Description:** The Gemini proxy (`server.js:231`) and OpenAI proxy (`server.js:529`) pass `req.body.model`, `req.body.contents`, `req.body.messages`, `req.body.prompt`, and `req.body.temperature` directly to upstream APIs without validation beyond checking for presence. An attacker with admin access could inject arbitrary parameter values (e.g., extremely large messages triggering high token costs, or invalid model names causing cryptic errors with stack traces exposed).
- **Root Cause:** No whitelist for `model` values. No maximum length check on `contents`/`messages`. No bounds check on `temperature`.
- **Severity:** **Medium**
- **Location:** `server.js:231-267`, `server.js:529-557`

### ISSUE-010: `processReferral` Swallows All Errors Silently
- **ID:** REL-003
- **Description:** `context/AuthContext.tsx:72-87` has a `try/catch` in `processReferral` that catches all errors and only logs to `console.error`. If the referral write fails (e.g., Firestore permissions error), the new user is created without the referral link, and the referrer loses their credit. The calling code in `createProfile` does not handle this failure either.
- **Root Cause:** Referral processing is treated as non-critical, but it has direct financial implications (referral earnings). Silent failure loses revenue attribution.
- **Severity:** **Medium**
- **Location:** `context/AuthContext.tsx:84-86`

### ISSUE-011: Firestore Rules Allow Any Signed-In User to Create `selar_pending` Documents
- **ID:** SEC-004
- **Description:** Firestore rule at `firestore.rules:123-131` allows any signed-in user to create documents in `selar_pending` with `plan in ["daily", "weekly", "monthly", "quarterly", "annual"]`. A malicious user could create documents with other users' UIDs (the rule requires `resource.data.uid == request.auth.uid`, but the Selar webhook processes based on `reference` match, not UID match). Combined with the webhook path, an attacker could pre-populate fake payment tokens.
- **Root Cause:** The `selar_pending` collection should only be writable via the backend (Firebase Admin), not directly by client SDK calls. The server-side `initiate` endpoint already creates these documents server-side (line 736), making client-side write rules unnecessary and dangerous.
- **Severity:** **Low** (requires social engineering to exploit, but a defensive gap)
- **Location:** `firestore.rules:123-131`

### ISSUE-012: Inconsistent Logging Practices
- **ID:** QUAL-001
- **Description:** The codebase mixes `console.log`/`console.error`/`console.warn` with Pino structured logger calls across backend modules. In `server.js` alone, lines 48-50 use `console.log`, lines 505-509 use `console.error`, while most of the file uses `logger.`. The `blogGenerator.js` uses `console.log` exclusively. `quantService.js` uses `logger.` but also `console.error` on line 334. This makes structured log aggregation (e.g., in Railway/Sentry) missing context and difficult to search.
- **Root Cause:** No eslint rule or code convention enforcing Pino as the sole logger. Gradual migration was incomplete.
- **Severity:** **Low**
- **Location:** Widespread — `server.js:48-50`, `server.js:505`, `backend/blogGenerator.js:370,384,387,397`, `backend/quantService.js:334`, and many more.

### ISSUE-013: `adminAuth` Middleware Applied to Gemini Proxy Blocks Regular Users
- **ID:** ARCH-002
- **Description:** The Gemini proxy at `server.js:231` uses `adminAuth` middleware, meaning only admins can make proxy calls. However, `services/gemini.ts` uses this proxy to generate predictions and analysis for the frontend. If the frontend is meant to use Gemini for any non-admin feature (test connection, generate content for logged-in users), those calls will fail with 401.
- **Root Cause:** The `adminAuth` middleware was likely over-applied during security hardening. The proxy exists to keep the API key server-side, but the auth gate is too restrictive if any user-facing AI features remain.
- **Severity:** **Medium** (blocks user-facing AI features if any exist; if intentional, this is documentation debt)
- **Location:** `server.js:231`

### ISSUE-014: No Retry Logic on Fapshi/Selar External HTTP Calls
- **ID:** REL-004
- **Description:** The Fapshi initiate (`server.js:636`), Fapshi verify (`server.js:693`), and Selar webhook processing make direct `fetch` calls to external payment APIs without retry logic. Unlike the quant pipeline which has `withExponentialBackoff`, payment operations have no resilience against transient network errors. A momentary Fapshi outage during payment verification loses the payment.
- **Root Cause:** Retry logic exists in `quantService.js` but was not generalized into a shared utility. Each HTTP call in `server.js` is bare `fetch` with no retries.
- **Severity:** **Medium**
- **Location:** `server.js:636,693`

### ISSUE-015: Missing Functional E2E Tests
- **ID:** TEST-002
- **Description:** All existing tests (`AuthHelpers.test.ts`, `db.test.ts`, `paymentPlans.test.js`) are unit tests for pure functions. There are no integration or end-to-end tests for:
  - Payment flows (initiate -> webhook -> fulfill)
  - VIP access gating (can non-VIP users read `quant_vip` collection?)
  - Admin trigger endpoints (auth + execution)
  - Firestore security rules (are the rules actually preventing unauthorized writes?)
  - Predictions data pipeline (fetch -> save -> read -> display)
  
  The only test in CI is the unit test suite (`vitest run`). All ad-hoc test scripts at root level (`test-api.js`, `test_backend_health.mjs`, etc.) are manual-run only.
- **Root Cause:** No testing strategy beyond basic unit tests. No Firestore rules simulator tests, no integration test harness, no API test suite in CI.
- **Severity:** **Medium**
- **Location:** `package.json:13-15` (test scripts), root-level ad-hoc test files

### ISSUE-016: `initScheduler` Not in Try Block — Unhandled Rejection
- **ID:** REL-005
- **Description:** In `server.js:87`, `initScheduler()` is called inside the `try` block for Firebase Admin initialization (line 73). If Firebase Admin fails to initialize (caught at line 107), the scheduler is never started. However, if `initScheduler()` itself throws synchronously (e.g., cron parsing error, import failure), the error is caught at line 107 and misinterpreted as a Firebase Admin failure, and the server continues without any cron jobs — with no alert beyond a generic Sentry exception.
- **Root Cause:** `initScheduler()` should be in its own try/catch with explicit failure handling, not piggybacked on Firebase initialization.
- **Severity:** **Low**
- **Location:** `server.js:73-110`

## 3. IMPROVEMENT STRATEGY

### SEC-001 (RevenueCat webhook auth): **Mandatory secret check**
Make the webhook secret mandatory. If `REVENUECAT_WEBHOOK_SECRET` is not configured, return 503 immediately. Never skip auth.

### TEST-001 (Stale payment test): **Update test expectations to match USD amounts**
Update `paymentPlans.test.js` to use the actual `PLAN_CONFIG` values: 4.99, 14.99, 24.99, 59.99, 99.99.

### SEC-002 (Admin rate limiting): **Add rate limiters to all admin endpoints**
Apply `adminLimiter` (e.g., 10 requests per 5 minutes per IP) to all admin trigger POST endpoints. Use a single shared limiter instance.

### REL-001 (Lock fallback): **Fail-closed on lock errors**
Change `acquireLock` to return `false` on Firestore errors, skipping the cron job with a warning. Add a separate "force run" override for emergencies.

### PAY-001 (Fapshi amount mismatch): **Add XAF-to-plan mapping**
Create a separate `inferPlanFromAmountXAF()` function or add a `currency` parameter to `inferPlanFromAmount`. Configure Fapshi plan amounts in XAF alongside USD amounts.

### ARCH-001 (Date-key inconsistency): **Extract a shared utility**
Create `backend/dateUtils.js` exporting `getLagosDateKey()` and import it in all backend modules. Use the same `toLocaleDateString` approach consistently.

### REL-002 (Python binary race): **Promise-based deduplication**
Use a pending promise pattern: if `_pythonBinPromise` is in-flight, await it instead of starting a new probe.

### OPS-001 (SW stale version): **Make build fail on regex mismatch**
Assert that the version replace succeeded. If the regex doesn't match, throw a build error.

### SEC-003 (Proxy input validation): **Whitelist and constrain inputs**
- Whitelist `model` against known valid model IDs.
- Enforce maximum `contents`/`messages` length (e.g., 100KB).
- Clamp `temperature` to [0, 2].
- Reject unexpected top-level keys.

### REL-003 (Referral error): **Surface referral failures**
Throw from `processReferral` on failure, and handle the error in `createProfile` by retrying or notifying the user.

### TEST-002 (Missing E2E): **Add integration test suite**
- Test payment flows with mocked Fapshi/Selar
- Test Firestore rules with the simulator
- Test admin endpoints with JWT auth
- Run integration tests in CI

## 4. IMPLEMENTATION PLAN

### Phase 1 — Critical Fixes (must deploy immediately)

**Step 1:** Fix RevenueCat webhook authentication (`server.js:834-841`)
  - Replace the conditional `if (REVENUECAT_WEBHOOK_SECRET)` with a mandatory check.
  - Return 503 if secret is not configured.
  - Add a startup-time assertion that logs a fatal error if the secret is missing.

**Step 2:** Fix Fapshi amount validation (`server.js:704-708`, `backend/paymentPlans.js`)
  - Add `XAF_PLAN_CONFIG` mapping XAF amounts to plan names in `paymentPlans.js`.
  - Add `inferPlanFromAmountXAF(amount)` function mapping XAF -> plan.
  - Update Fapshi verify to use `inferPlanFromAmountXAF()` instead of `inferPlanFromAmount()`.

**Step 3:** Fix stale payment plan tests (`backend/paymentPlans.test.js`)
  - Update all amount assertions to match `PLAN_CONFIG` (USD values: 4.99, 14.99, 24.99, 59.99, 99.99).
  - Run `npm test` to verify all tests pass.

### Phase 2 — High Priority (deploy within 48 hours)

**Step 4:** Add rate limiting to admin trigger endpoints (`server.js`)
  - Create a shared `adminLimiter` with 10 requests per 5-minute window per IP.
  - Apply to routes: `/api/admin/generate-basketball`, `/api/admin/generate-cricket`, `/api/admin/generate-blog`, `/api/admin/generate-accumulators`, `/api/admin/telegram-broadcast`, `/api/admin/trigger-quant`, `/api/admin/grade-quant`, `/api/admin/quant-performance`.
  - Also apply to `/api/admin/token` (currently has no limiter at all).

**Step 5:** Fix distributed lock fail-open behavior (`backend/scheduler.js:46-48`)
  - Change `return true` to `return false` in the catch block of `acquireLock`.
  - Log a warning: `cron job {taskName} skipped — Firestore lock acquisition failed`.
  - Add a `/api/admin/force-run/:taskName` endpoint for manual override.

**Step 6:** Extract shared date-key utility
  - Create `backend/dateUtils.js` with `export const getLagosDateKey = () => new Date().toLocaleDateString('en-CA', { timeZone: 'Africa/Lagos' })`.
  - Replace manual implementations in `blogGenerator.js` (lines 3-10) and `telegramService.js` (lines 23-30).
  - Verify the `quantService.js` and `db.ts` implementations match; replace with import if accessible.

### Phase 3 — Medium Priority (deploy within 1 week)

**Step 7:** Add input validation to Gemini/OpenAI proxy endpoints (`server.js`)
  - Whitelist `model` against known model IDs for Gemini (`gemini-2.5-flash`, `gemini-2.0-flash`, `gemini-2.5-pro`, `gemini-2.0-flash-exp`) and OpenAI (`gpt-4o-mini`, `gpt-4o`, `gpt-4.1-nano`).
  - Enforce max `contents` string length of 50KB and max `messages` array total string length of 50KB.
  - Clamp `temperature` to [0, 2]; default to 0.7 if missing.
  - Strip unexpected keys before forwarding to upstream API.

**Step 8:** Fix Python binary resolution race condition (`backend/quantService.js`)
  - Introduce `let _pythonBinPromise = null` alongside `_pythonBin`.
  - In `resolvePythonBin()`, if `_pythonBin`, return it; if `_pythonBinPromise`, await it; otherwise set `_pythonBinPromise = probeAllCandidates()` and store result before returning.

**Step 9:** Add retry logic to Fapshi/Selar HTTP calls (`server.js`)
  - Extract the `withExponentialBackoff` utility from `quantService.js` into `backend/retryUtils.js`.
  - Wrap Fapshi fetch at line 636, Fapshi status fetch at line 693, and Selar webhook processing fetch calls with 2 retries and 2s base delay.

**Step 10:** Surface referral processing errors (`context/AuthContext.tsx`)
  - Change `processReferral` to re-throw after logging.
  - In `createProfile` on line 184, add a `.catch()` that shows a toast notification: "Referral code could not be applied. Please try again later."

### Phase 4 — Low Priority (deploy within 2 weeks)

**Step 11:** Make Service Worker version replacement fail on mismatch (`vite.config.ts`)
  - After the `replace()` call, check if the new version string appears in the file.
  - If not, throw a build error: `[SW Version Plugin] Version string replacement failed — CACHE_VERSION format may have changed.`

**Step 12:** Standardize logging to Pino (`server.js`, `blogGenerator.js`, `quantService.js`)
  - Replace all `console.log` / `console.error` / `console.warn` in backend modules with `logger.info` / `logger.error` / `logger.warn`.
  - Add an ESLint rule `no-console` (with exceptions for frontend code) to enforce this.

**Step 13:** Restrict `selar_pending` Firestore write rules (`firestore.rules`)
  - Change `allow create` on `selar_pending` to `if false` (or `if isAdmin()` for admin overrides).
  - The backend already writes to this collection server-side; client-side writes are unnecessary.

**Step 14:** Move `initScheduler()` to its own try/catch (`server.js`)
  - Extract the `initScheduler()` call from the Firebase init try block (line 87).
  - Place in a separate try/catch that logs a fatal error and sends a Sentry alert if scheduling fails.

**Step 15:** Add integration test infrastructure
  - Install `@firebase/rules-unit-testing` for Firestore rules tests.
  - Create `backend/__tests__/payment-flow.test.js` with mocked Fapshi/Selar HTTP responses.
  - Create `backend/__tests__/admin-routes.test.js` with JWT token tests.
  - Add `npm run test:integration` script to `package.json`.
  - Add integration tests to CI pipeline (Codemagic or Netlify build).
